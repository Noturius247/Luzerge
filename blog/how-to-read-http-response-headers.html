How to Read HTTP Response Headers for Security and Performance

Luzerge Team March 13, 2026 7 min read

Every time your browser loads a web page, the server sends back more than just HTML. It sends a set of HTTP response headers — invisible metadata that tells the browser how to handle the content. These headers control everything from caching behavior and content type to critical security policies that protect your visitors from attacks.

Most website owners never look at their response headers. That is a problem, because misconfigured or missing headers can leave your site vulnerable to cross-site scripting, clickjacking, and data leakage — even if your application code is perfectly secure.

In this guide, we will break down the most important response headers, explain what good values look like, and show you how to inspect them using free tools including Luzerge's Header Inspector.

What Are HTTP Response Headers?

When a browser requests a URL, the server responds with two parts: the headers and the body. The body is the content you see — the HTML, CSS, images, or JSON data. The headers are key-value pairs that provide instructions to the browser about how to process that content.

For example, a typical response might include headers like:

  • Content-Type: text/html; charset=UTF-8 — tells the browser this is an HTML page
  • Cache-Control: max-age=3600 — tells the browser to cache this page for one hour
  • Strict-Transport-Security: max-age=31536000 — tells the browser to always use HTTPS

Headers fall into two broad categories that matter most for website operators: security headers that protect against attacks, and cache/performance headers that control how content is stored and delivered.

HTTP Response Headers
HTTP/1.1 200 OK content-type: text/html; charset=UTF-8 cf-cache-status: HIT cf-ray: 8abc1234-SIN strict-transport-security: max-age=31536000; includeSubDomains content-security-policy: default-src 'self' x-content-type-options: nosniff x-frame-options: DENY referrer-policy: strict-origin-when-cross-origin

Security Headers Explained

Security headers are your website's first line of defense at the protocol level. They instruct browsers to enforce security policies before any JavaScript or application logic runs. Missing security headers is one of the most common findings in web security audits.

Strict-Transport-Security (HSTS)

HSTS tells the browser: "Always connect to this site over HTTPS, even if the user types http:// in the address bar." Without HSTS, an attacker on the same network could intercept the initial HTTP request before it redirects to HTTPS — a classic man-in-the-middle attack.

Recommended value: Strict-Transport-Security: max-age=31536000; includeSubDomains; preload

The max-age=31536000 tells the browser to remember this policy for one year. The includeSubDomains directive extends the policy to all subdomains, and preload allows your domain to be included in browsers' built-in HSTS preload lists. If you see max-age=0 or this header missing entirely, the site has no HSTS protection.

Content-Security-Policy (CSP)

CSP is arguably the most powerful security header available. It defines exactly which sources of content are allowed to load on your page — scripts, styles, images, fonts, and more. A well-configured CSP stops cross-site scripting (XSS) attacks dead in their tracks by preventing the browser from executing unauthorized scripts.

Example value: Content-Security-Policy: default-src 'self'; script-src 'self' https://cdn.example.com; style-src 'self' 'unsafe-inline'

This policy says: only load resources from the same origin by default, only allow scripts from the site itself and a specific CDN, and allow inline styles. If an attacker injects a script tag pointing to their malicious server, the browser will refuse to load it. A missing CSP header means any script from any source can execute on your page.

X-Content-Type-Options

This header prevents MIME sniffing — a browser behavior where it tries to guess the content type of a file instead of trusting the declared Content-Type header. Attackers exploit MIME sniffing by uploading malicious files disguised as harmless ones (like an HTML file pretending to be an image).

Recommended value: X-Content-Type-Options: nosniff

There is only one valid value for this header. If it is missing, browsers may interpret files in dangerous ways. This is one of the easiest security headers to add — there is no configuration to think about.

X-Frame-Options

This header prevents your website from being embedded inside an <iframe> on another site. Without it, attackers can overlay your page with a transparent iframe and trick users into clicking hidden buttons — a technique called clickjacking.

Recommended value: X-Frame-Options: DENY or X-Frame-Options: SAMEORIGIN

Use DENY if your site never needs to be embedded anywhere. Use SAMEORIGIN if you embed your own pages in iframes (for example, in an admin panel). The value ALLOW-FROM is deprecated and should not be used.

Referrer-Policy

When a visitor clicks a link on your site that leads to an external page, the browser sends a Referer header to the destination site telling them where the visitor came from. This can leak sensitive information — especially if your URLs contain tokens, user IDs, or internal paths.

Recommended value: Referrer-Policy: strict-origin-when-cross-origin

This sends only the origin (domain) to external sites, the full URL to same-origin requests, and nothing when downgrading from HTTPS to HTTP. If this header is missing, the browser uses its default policy, which may share more information than you intend.

Cache and Performance Headers

Cache headers determine how long browsers and CDNs store your content. Correct caching dramatically improves page speed and reduces server load. Incorrect caching can serve stale content or prevent updates from reaching visitors.

Cache-Control

This is the primary header that controls caching behavior across browsers and CDN edge servers. It supports multiple directives:

  • max-age=3600 — cache the resource for 3600 seconds (1 hour)
  • public — the response can be cached by any cache, including CDNs
  • private — only the end user's browser may cache this (not CDN edges)
  • no-cache — the cache must revalidate with the origin before serving (does not mean "don't cache")
  • no-store — do not cache this response at all
  • s-maxage=86400 — CDN-specific max-age, overrides max-age for shared caches

Good value for static assets: Cache-Control: public, max-age=31536000, immutable — cache for one year, never revalidate. Use this for versioned assets like app.a1b2c3.js.

Good value for HTML pages: Cache-Control: public, max-age=0, must-revalidate — always check for updates but use cache if nothing has changed.

CF-Cache-Status

If your site is behind Cloudflare, this header tells you whether the response was served from Cloudflare's edge cache or fetched from your origin server. Understanding these values is key to diagnosing performance issues:

  • HIT — the resource was served from Cloudflare's cache. This is the ideal state for static assets — fastest possible delivery, zero origin load.
  • MISS — the resource was not in cache and was fetched from your origin. The next request should be a HIT.
  • DYNAMIC — Cloudflare considers this resource ineligible for caching (typically HTML pages or API responses). You can override this with Page Rules or Cache Rules.
  • EXPIRED — a cached copy existed but its TTL expired, so Cloudflare fetched a fresh copy from origin.
  • REVALIDATED — Cloudflare checked with origin and the cached version was still valid.

If your static assets consistently show DYNAMIC or MISS, your caching is misconfigured. Check your Cache-Control headers and Cloudflare cache rules.

CF-Ray

This header contains a unique identifier for every request that passes through Cloudflare's network. The format is a hex string followed by a dash and a three-letter airport code representing the data center that handled the request — for example, CF-Ray: 7a1b2c3d4e5f-MNL indicates the request was processed through Cloudflare's Manila data center.

The CF-Ray ID is essential for debugging. When you file a support ticket with Cloudflare or need to trace a specific request through your logs, this ID lets you pinpoint exactly what happened.

Age

The Age header tells you how many seconds the response has been sitting in the cache. If you see Age: 3600 alongside CF-Cache-Status: HIT, it means the cached copy is one hour old. This helps you verify that your TTL settings are working correctly and understand how fresh the content your visitors are seeing actually is.

How to Check Your Headers

There are three main ways to inspect HTTP response headers:

1. Browser Developer Tools

In Chrome or Firefox, press F12 to open developer tools, go to the Network tab, and reload the page. Click any request to see its response headers in the right panel. This is quick and convenient but only shows headers for pages you can visit in a browser.

2. Command Line with curl

Run curl -I https://example.com in your terminal to see only the response headers without downloading the body. The -I flag sends a HEAD request. For a full GET request showing headers, use curl -v https://example.com. This method is ideal for scripting and automation.

Good Values vs. Bad Values

Here is a quick reference for what healthy headers look like compared to problematic ones:

  • HSTS: Good: max-age=31536000; includeSubDomains. Bad: missing entirely, or max-age=86400 (too short to provide meaningful protection).
  • CSP: Good: a specific policy listing allowed sources. Bad: missing entirely, or default-src * which allows everything and provides no protection.
  • X-Content-Type-Options: Good: nosniff. Bad: missing (there is no other valid value).
  • Cache-Control for static assets: Good: public, max-age=31536000. Bad: no-cache, no-store on assets that should be cached, forcing re-download on every visit.
  • CF-Cache-Status for CSS/JS: Good: HIT. Bad: consistently DYNAMIC or MISS, meaning Cloudflare is not caching your static files.
A website with missing security headers is like a house with unlocked doors — it might be fine today, but you are inviting trouble.

How to Fix Missing Security Headers in Cloudflare

If your header inspection reveals missing security headers, Cloudflare makes it straightforward to add them without modifying your origin server. Here is how:

Method 1: Transform Rules (Recommended)

  1. Log in to your Cloudflare dashboard and select your domain
  2. Navigate to Rules > Transform Rules > Modify Response Header
  3. Click Create rule
  4. Set the filter to match all requests (or specific paths)
  5. Add each header as a "Set static" action:
    • X-Content-Type-Options = nosniff
    • X-Frame-Options = DENY
    • Referrer-Policy = strict-origin-when-cross-origin
    • Strict-Transport-Security = max-age=31536000; includeSubDomains
  6. Click Deploy

Method 2: Cloudflare Workers

For more complex configurations like Content-Security-Policy, a Cloudflare Worker gives you full control. You can intercept every response and add or modify headers programmatically. This approach is best when your CSP needs to vary by page or include dynamic nonces for inline scripts.

After Deploying

Once you have added your security headers, verify them by running your site through the Luzerge scanner again. Check that every security header appears in the response and that the values match your intended configuration. It is also good practice to test on a staging domain first if you are adding a strict Content-Security-Policy, as overly restrictive rules can break page functionality.

Response headers are one of the most overlooked aspects of web security and performance. They take minutes to configure but provide lasting protection against common attacks and meaningful performance improvements through proper caching. Take seven minutes today to inspect your headers — you might be surprised by what you find.

Frequently Asked Questions

What are HTTP response headers?
HTTP response headers are metadata sent by a web server alongside the actual page content. They tell the browser how to handle the response — including caching rules, security policies, content type, and server identity. You can view them in browser developer tools or by using command-line tools like curl.
What security headers should my website have?
Every website should have at minimum: Strict-Transport-Security (HSTS) to enforce HTTPS, Content-Security-Policy (CSP) to prevent XSS attacks, X-Content-Type-Options set to 'nosniff' to prevent MIME sniffing, X-Frame-Options set to 'DENY' or 'SAMEORIGIN' to prevent clickjacking, and Referrer-Policy to control information leakage to third-party sites.
What does CF-Cache-Status HIT mean?
CF-Cache-Status HIT means Cloudflare served the resource directly from its edge cache without contacting your origin server. This is the ideal status for static assets because it means faster load times for visitors and reduced load on your server. Other values include MISS (fetched from origin), DYNAMIC (not eligible for caching), and EXPIRED (cached copy was stale and re-fetched).
How do I add security headers to my website?
If you use Cloudflare, the easiest way is through Transform Rules. Go to your Cloudflare dashboard, navigate to Rules > Transform Rules > Modify Response Header, and add each security header with its recommended value. You can also set headers at the server level using your web server configuration (Apache .htaccess, Nginx config) or through your application framework's middleware.
