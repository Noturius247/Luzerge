Understanding CDN Cache Hit Rates and How to Improve Them

Luzerge Team March 15, 2026 6 min read

Every time a visitor loads your website, the browser makes dozens of requests for HTML, CSS, JavaScript, images, and fonts. If those assets are served from a CDN edge server close to the visitor, the page loads fast. If every request has to travel all the way back to your origin server, the page loads slowly and your server bears the full cost of every single request.

The metric that captures this difference is your cache hit rate — and it is one of the most important numbers you should be tracking for website performance.

What Is a Cache Hit Rate and Why It Matters

Your cache hit rate is the percentage of total requests that are served directly from the CDN's edge cache, without needing to contact your origin server. The formula is straightforward:

Cache Hit Rate = (Requests Served from Cache / Total Requests) x 100%

If your site receives 10,000 requests in a day and 9,200 of them are served from cache, your cache hit rate is 92%. The remaining 800 requests went back to your origin server — these are called cache misses.

Why does this matter? Three reasons:

  • Speed — A cache hit is served from a server geographically close to the visitor, often in under 20 milliseconds. A cache miss might take 200-500ms or more, depending on how far the origin server is. For visitors in the Philippines hitting a US-hosted origin, that round trip adds significant latency.
  • Bandwidth costs — Every cache miss means your origin server has to process the request and transfer the data. If you are on a hosting plan with bandwidth limits, a low cache hit rate burns through your allocation faster. CDN-served requests typically do not count against your origin bandwidth.
  • Origin server load — The fewer requests that reach your origin, the less CPU, memory, and I/O it needs to handle. A high cache hit rate means your origin server can handle traffic spikes without crashing, because the CDN absorbs most of the load.

How Cloudflare's CDN Caching Works

Cloudflare operates a global network of over 300 data centers, including nodes in Manila, Cebu, and other locations across Asia. When you put your site behind Cloudflare, here is what happens for each request:

  1. Visitor makes a request — their browser sends a request for a resource, say /img/hero.webp.
  2. Request hits the nearest edge server — Cloudflare routes the request to the closest data center using Anycast. For a visitor in Manila, that would be the Manila edge node.
  3. Edge checks its cache — if the edge server already has a fresh copy of hero.webp, it serves it directly. This is a cache hit.
  4. Cache miss triggers an origin pull — if the edge does not have the file (first request, expired cache, or the file was never cached), Cloudflare fetches it from your origin server. It then caches the response at the edge for future requests.

By default, Cloudflare caches static assets based on file extension: images (.jpg, .png, .webp), stylesheets (.css), scripts (.js), fonts (.woff2), and other common static file types. It does not cache HTML pages by default — those are treated as dynamic content unless you explicitly tell Cloudflare otherwise.

Understanding CF-Cache-Status Headers

Cloudflare adds a CF-Cache-Status header to every response, telling you exactly what happened with caching for that request. Understanding these values is essential for diagnosing cache performance issues. You can inspect these headers using your browser's developer tools or by reading our guide to HTTP response headers.

Here are the most common values:

  • HIT — The resource was found in Cloudflare's edge cache and served directly. This is what you want to see for static assets.
  • MISS — The resource was not in the edge cache, so Cloudflare fetched it from your origin. The response is now cached for future requests. You will typically see MISS on the first request to a particular edge node.
  • EXPIRED — The resource was in the cache but its TTL (time to live) had expired. Cloudflare fetched a fresh copy from the origin. If your TTL values are too short, you will see EXPIRED frequently, which hurts your cache hit rate.
  • DYNAMIC — Cloudflare determined that this resource should not be cached. This is the default for HTML pages, API endpoints, and anything served with Cache-Control: no-cache or private headers. DYNAMIC responses are always fetched from the origin.
  • REVALIDATED — The cached resource was stale, but Cloudflare checked with the origin (using If-Modified-Since or If-None-Match) and the origin confirmed the cached version is still valid. This saves bandwidth because only headers are exchanged, not the full file.
  • BYPASS — Caching was explicitly bypassed, usually by a Cloudflare Page Rule or Worker that sets Cache-Control: bypass.

What Is Tiered Caching and How It Reduces Origin Load

Without tiered caching, every Cloudflare edge node that gets a cache miss will independently contact your origin server. If your site gets traffic from 50 different Cloudflare data centers, a cold cache scenario means 50 separate origin pulls for the same resource.

Tiered caching adds a middle layer. Instead of every edge node going directly to your origin, cache misses are first routed to a larger, regional "upper-tier" data center. If that upper-tier node has the resource cached, it serves it to the edge — and your origin never sees the request.

The result is significant:

  • Fewer origin pulls — instead of 50 edge nodes each hitting your origin, only a handful of upper-tier nodes do. Origin requests can drop by 70-90%.
  • Better cache hit rates at the edge — because upper-tier nodes aggregate traffic from multiple edge nodes, they build and maintain cache more efficiently.
  • Faster cache fills — when an edge node misses, it fetches from a nearby upper-tier node instead of a potentially distant origin server. The edge-to-upper-tier connection is optimized and faster than edge-to-origin.

Tiered caching is available on all Cloudflare plans, including the free plan. You can enable it in the Cloudflare dashboard under Caching > Tiered Cache. For a full walkthrough, see our Cloudflare setup guide.

Five Practical Tips to Improve Your Cache Hit Rate

1. Set Proper Cache-Control Headers

Your origin server's Cache-Control headers tell Cloudflare how long to cache each resource. If you are not setting these headers, Cloudflare falls back to its default behavior, which may not be optimal for your site.

For static assets like images, CSS, and JavaScript files, set long cache durations. A common configuration is:

  • Static assets (images, CSS, JS, fonts): Cache-Control: public, max-age=31536000, immutable — one year. Use versioned filenames or query strings to bust the cache when files change.
  • HTML pages (if you want them cached): Cache-Control: public, max-age=3600, s-maxage=86400 — the s-maxage directive tells CDNs to cache for 24 hours, while browsers cache for 1 hour.
  • API responses: Usually Cache-Control: no-store or private unless the responses are truly static.

The s-maxage directive is particularly useful because it lets you set different cache durations for CDN edge servers versus end-user browsers.

2. Use Cloudflare Page Rules for Aggressive Caching

By default, Cloudflare does not cache HTML. If your site is mostly static — a blog, a portfolio, a landing page — you are leaving performance on the table. Cloudflare Page Rules let you override the default behavior for specific URL patterns.

Create a Page Rule for your static pages with Cache Level: Cache Everything and set an Edge Cache TTL of 1-7 days. This tells Cloudflare to cache HTML responses at the edge, dramatically improving load times for returning visitors and reducing origin load.

Be careful with this on pages that have user-specific content (dashboards, account pages). Only use "Cache Everything" on truly static or public pages.

3. Minimize Query String Variations

Cloudflare treats each unique URL as a separate cache entry. That means /page?utm_source=facebook and /page?utm_source=twitter are cached as two different resources, even though the content is identical.

Marketing campaign URLs with UTM parameters, session IDs in query strings, and analytics tracking parameters all fragment your cache. Each variation creates a new cache key, which means more cache misses and more origin pulls.

Solutions include:

  • Use Cloudflare's Cache Key settings to ignore specific query string parameters
  • Strip tracking parameters on the server side before they reach the cache layer
  • Move tracking to JavaScript-based solutions that do not modify the URL

4. Enable Tiered Caching

As discussed above, tiered caching is one of the simplest and most impactful changes you can make. It reduces origin pulls significantly by adding a shared cache layer between edge nodes and your origin. If you have not enabled it yet, do so now — it is free and takes less than a minute.

Go to your Cloudflare dashboard, navigate to Caching > Tiered Cache, and select Smart Tiered Caching Topology. Cloudflare will automatically route cache misses through the optimal upper-tier nodes for your traffic patterns.

5. Purge Cache Strategically, Not Blanket Purges

When you update your site, you need to purge the old cached content. Many site owners reach for the "Purge Everything" button in Cloudflare. This works, but it is a sledgehammer approach. Every single cached asset across all edge nodes is cleared, and every subsequent request becomes a cache miss until the cache is rebuilt.

Instead, use targeted cache purges:

  • Purge by URL — only clear the specific files you updated. Changed your homepage? Purge /index.html and nothing else.
  • Purge by tag — if you use Cache-Tag headers (available on Business and Enterprise plans), you can purge all resources tagged with a specific label, like blog-posts or product-images.
  • Purge by prefix — clear all cached content under a specific path, like /blog/.

By purging only what changed, you keep the rest of your cache warm and your hit rate high. A blanket purge after every deploy can tank your cache hit rate for hours.

Frequently Asked Questions

What is a good cache hit rate?
A good cache hit rate is generally above 90%, and an excellent one is above 95%. For mostly static websites like blogs, landing pages, and portfolios, you should aim for 95% or higher. Dynamic sites with personalized content may see lower rates, but you can still push cacheable assets above 90% with the right Cache-Control headers and page rules.
Why is my cache hit rate low?
Common causes of low cache hit rates include: missing or incorrect Cache-Control headers on your origin server, too many query string variations creating unique cache keys, frequent full cache purges that clear everything unnecessarily, short TTL values that cause assets to expire too quickly, and not having tiered caching enabled — which means fewer edge nodes share cached content.
What does CF-Cache-Status: DYNAMIC mean?
CF-Cache-Status: DYNAMIC means Cloudflare determined the resource is not eligible for caching based on its default rules. This typically applies to HTML pages, API responses, and any content served with Cache-Control: no-cache or private headers. You can override this behavior using Cloudflare Page Rules with "Cache Everything" to force caching of specific URLs.
Does caching affect SEO?
Yes, caching positively affects SEO. Faster page load times are a confirmed Google ranking factor, and a high cache hit rate directly reduces load times by serving content from edge servers close to the visitor. Additionally, caching reduces origin server load, which means your server can handle Googlebot crawls more efficiently without slowdowns.
